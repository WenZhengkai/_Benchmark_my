## Module Name
EXU
### Overview
EXU is the core module responsible for executing instructions in the processor. It receives instruction information from the instruction issuance unit (ISU) and distributes it to different functional units (ALU, LSU, CSR) for processing according to the instruction type. After processing, the result is passed to the write back unit (WBU). At the same time, EXU is also responsible for processing branch jump instructions and generating redirection signals to correct the PC value of the prediction error.

### Parameters
This module inherits from `NPCModule` and `HasNPCParameter`, and uses the `TYPE_INST` trait, indicating that it is related to the processor core parameters and instruction types. The specific parameters depend on the definition of the parent class and trait.

### Internal logic

1. **Output Connection**
In IO port, between from_isu and to_wbu, cf/ctrl/data should be connected first

2. **Functional unit distribution**:
Three functional unit modules need to be instantiated, ALU, LSU, CSR
All three require similar wiring operations, connecting 2 operands, operation type number, instantiate alu0 for example, ensure srca, srcb, fuOpType, ready, valid are connected
alu.io.in.bits.srca := io.from_isu.bits.data.fuSrc1 // fuSrc2 is the same
alu.io.in.bits.fuOpType := io.from_isu.bits.ctrl.fuOpType
alu.io.out.ready := io.to_wbu.ready
alu0.io.in.valid := io.from_isu.valid && (io.from_isu.bits.ctrl.fuType === FuType.alu)
io.to_wbu.bits.data.Alu0Res.bits := alu0.io.out.bits
The rest of the modules require similar wiring, and there may be other ports connected
- **ALU**: Processing arithmetic and logical operation instructions. According to `fuType`, it determines whether it is an ALU instruction, and passes the operand and operation type to the ALU module.
- **LSU**: Processing load and store instructions. Also according to `fuType`, it determines whether it is an LSU instruction, and passes the operand and operation type to the LSU module. The LSU module is also responsible for interacting with the memory unit.
- lsu input
Note the port signals, from_mem, to_mem, ctrl, data as input, connected to:
lsu0.io.from_mem
lsu0.io.ctrl
lsu0.io.data
- lsu output
lsu0.io.out.bits, lsu0.io.to_mem as output, connected to:
io.to_wbu.bits.data.data_from_mem
io.to_mem
- **CSR**: Process control and status register instructions. Determine whether it is a CSR instruction based on `fuType`, and call the access interface of the CSR module.
The output result is connected to io.to_wbu.bits.data.csrRdata

3. **Branch jump processing**:
The following signal bundles were previously defined to wrap the signals about the jump results (no need to repeat the definition)
class BruRes extends NPCBundle {
val valid = Bool() // Is the signal bundle valid, that is, whether the corresponding jump instruction is detected
val targetPc = UInt(XLen.W) // PC jump target address
}
Use Wire() to instantiate a bruRes object and call the method CalBruRes(), bruRes := CalBruRes(inBits, alu0, csr0)
At this time, bruRes carries valid signals and target addresses

- Generate redirection signal `redirect`, when the predicted PC value (next_pc of the port) is inconsistent with the actual calculated target PC value, it is defined as a prediction error.
The redirect.valid signal is determined by from_isu.valid, bruRes.valid, and the prediction error. When the three are true, valid is valid, and target is set to the target address.
- Update `out.bits.cf.next_pc` to the correct target PC value. If the prediction error does not occur, no change is required.

4. **Handshake signal processing**:
- Use the `HandShakeDeal` function to process the input and output `ready/valid` signals.
Call the following function
HandShakeDeal(io.from_isu, io.to_wbu,
AnyInvalidCondition = false.B,
AnyStopCodition = io.redirect.valid && in.valid
)
