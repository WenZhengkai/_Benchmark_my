## Module-WBU (Write Back Unit)

### Overview
The WBU module is the write-back stage in the processor pipeline, responsible for writing the results of the execution stage (EXU) back to the register file. The module receives data from the EXU, selects the correct write-back data source based on the control signal, and passes the data to the register file module through the handshake protocol.

### Parameters
No explicit parameters, but depends on the configuration defined in the parent class NPCModule and NPCBundle (such as XLen for data bit width).

### Input/Output Interface
#### Input Interface
- `from_exu`: Decoupled interface from the execution unit, including:
- `ctrl`: control signal (rd register number, rfWen register write enable, ResSrc result selection signal)
- `data`: data signal (ALU result, memory read data, CSR read data)
- `cf`: control flow information (passed to the commit stage)

#### Output Interface
- `to_reg`: Decoupled interface to the register file, including:
- `rd`: target register number (5 bits)
- `Res`: data to be written (XLen bit)
- `RegWrite`: register write enable signal
- `to_commit`: control flow information output to the commit stage

   val to_reg = Decoupled(new WbuToRegIO) 
   val to_commit = Output(new CtrlFlow) //Output the control flow information to the submission phase 
})

### Task 1: Handshake Protocol Implementation
**Objective:**
Implement the handshake protocol to ensure data transmission from `io.from_exu` to `io.to_reg` respects the ready/valid handshake.

**Steps:**
1. Define a `HandShakeDeal` function or utility to handle the handshake.
2. Ensure `io.from_exu.ready` is asserted when `io.to_reg.ready` is asserted and the handshake conditions are valid.
3. Pass `AnyInvalidCondition` (initially hardcoded as `false.B`, future extension possible) into the handshake logic.
4. Integrate the handshake into the main WBU logic by invoking `HandShakeDeal(io.from_exu, io.to_reg, AnyInvalidCondition)`.

### Task 2: Data Selection Logic for Write-Back
**Objective:**
Implement logic to select the correct data source (ALU result, memory read data, or CSR read data) to write back to the register file based on the `ResSrc` signal.

**Steps:**
1. Decode the value of the control signal `ResSrc`.
2. Create a multiplexer using Chisel to decode the value:

   - If `ResSrc === 1.U`, select `data.data_from_mem`.
   - If `ResSrc === 2.U`, select `data.csrRdata`.
   - Set default value to `0.U` if `ResSrc` is invalid or unhandled.
3. Handle corner cases, e.g., add default cases to avoid unintended behaviors.
4. Write the selected data to the `to_reg.Res` output.

---

Generate all output interface signals (`to_reg`, `to_commit`) and ensure they are set up correctly based on the inputs.

1. Pass the `rd` signal directly from `io.from_exu.ctrl.rd` to `to_reg.rd`.
2. Assign the write enable signal (`to_reg.RegWrite`) as the AND gate result of:
   - `io.from_exu.ctrl.rfWen`
   - `io.to_reg.valid`.
3. Ensure the `to_commit` output is passed directly from `io.from_exu.cf` to maintain the control flow context.
4. Check and maintain valid/ready signals across outputs (`io.to_reg.valid` especially).
