## Module-EXU (Execution Unit)

### Overview
EXU is the core module responsible for executing instructions in the processor. It receives instruction information from the instruction issuance unit (ISU) and distributes it to different functional units (ALU, LSU, CSR) for processing according to the instruction type. After processing, the result is passed to the write back unit (WBU). At the same time, EXU is also responsible for processing branch jump instructions and generating redirection signals to correct the PC value of the prediction error.

### Parameters
This module inherits from `NPCModule` and `HasNPCParameter`, and uses the `TYPE_INST` trait, indicating that it is related to the processor core parameters and instruction types. The specific parameters depend on the definition of the parent class and trait.

### Input/output interface

## Module - ALU

### Overview
The ALU module implements a core processor component that supports a variety of arithmetic and logical operations. It performs basic arithmetic operations (addition and subtraction), bitwise operations (AND/OR/XOR), shifts, and comparisons. It also supports branch decision logic and 32-bit/64-bit word length operations.

### Parameters
- XLen: Data width (inherited from NPCBundle, typically 32 or 64 bits)

### Input/Output Interfaces
class FunctionUnitOut extends NPCBundle {
    val out = Decoupled(UInt(XLen.W))
}

class FunctionUnitIO extends FunctionUnitOut {
    val in = Flipped(Decoupled(new Bundle {
        val srca = Output(UInt(XLen.W))
        val srcb = Output(UInt(XLen.W))
        val fuOpType = Output(FuOpType())
    }))
    
}

class ALUIO extends FunctionUnitIO {
    val taken = Output(Bool())      // if Branch Taken
}

### Internal Logic

1. **Basic Arithmetic Unit:**
- Adder/Subtractor: Unified addition and subtraction processing using two's complement
- XOR Unit: Used for comparison and logical operations
- Shifter: Supports logical left/right shifts and arithmetic right shifts
- Comparator: Generates the slt/sltu signals from the adder result

2. **Operation Type Decoding:**
- Uses the encoding defined by the ALUOpType object to identify the current operation
- Supported operations include:
* Arithmetic: add/sub/addw/subw
* Logical: and/or/xor
* Shift: sll/srl/sra/sllw/srlw/sraw
* Comparison: slt/sltu
* Branch: beq/bne/blt/bge/bltu/bgeu
* Jump: jal/jalr/call/ret

3. **Word Length Handling:**
- Identifies 32-bit operations using isWordOp
- Sign-extends the 32-bit result

4. **Branch Decision Logic:**
- Generates a taken signal based on the operation type and comparison result.
- Supports multiple branch conditions such as beq/bne/blt/bge/bltu/bgeu

## Module - LSU 

### Overview
The LSU module is the functional unit in the processor responsible for processing load and store instructions. It interacts with the memory subsystem, handles load/store operations of varying bit widths, including signed and unsigned extension, and supports basic atomic operations.

### Parameters
- `XLen`: Data width (provided via the HasNPCParameter trait)
- Supported operation types are defined in the LSUOpType object, including:
- Load instructions: lb, lh, lw, ld, lbu, lhu, lwu
- Store instructions: sb, sh, sw, sd
- Atomic operations: lr, sc, amoswap, amoadd, etc.

### Input/Output Interface
class LSUIO extends FunctionUnitIO {
    val to_mem = new ToMem
    val from_mem = new FromMem
    val ctrl    = Input(new CtrlSignal)
    val data  = Input(new DataSrc)
}

### Internal Logic
1. **Address Calculation**:
- The memory address is calculated by adding the input operands srca and srcb: `io.to_mem.addr := io.in.bits.srca + io.in.bits.srcb

2. **Write Mask Generation**:
- Generates different write masks depending on the operation type:
- sb (store byte): "b00"
- sh (store halfword): "b01"
- sw (store word): "b10"
- sd (store doubleword): "b11"

3. **Data Load Processing**:
- Processes memory return data based on the load instruction type:
- Signed loads (lb, lh, lw): Sign extension (SignExt) is used
- Unsigned loads (lbu, lhu, lwu): Zero extension (ZeroExt) is used
- Data read from memory is split and extended to the XLen bit width

4. **Control Signal Transfer**:
- Memory write enable signal is directly transferred: `io.to_mem.MemWrite := io.ctrl.MemWrite`
- Output valid signal is directly transferred to input valid: `io.out.valid := `io.in.valid`
- Input ready signal is fixed to true: `io.in.ready := true.B`

## Module - CSR (Control and Status Register)

### Overview
The CSR module is the core module for managing and controlling processor status in the RISC-V processor. It handles read and write operations of the Control and Status Register (CSR), exception handling, and privileged mode switching. This module implements the machine mode (M-mode)-related CSRs defined in the RISC-V privileged architecture and supports basic CSR operation instructions and exception handling mechanisms.

### Parameters
- `XLen`: Data width (32 or 64 bits), determines register width and the initial value of certain CSRs

### Input/Output Interface
class CSRIO extends FunctionUnitIO {
  val cfIn  = Input(new CtrlFlow)
  val jmp   = Output(Bool())

}

### Internal Logic
1. **CSR Register Group**:
- `mtvec`: Machine-mode Trap Vector Base Register
- `mcause`: Machine-mode Trap Cause Register
- `mepc`: Machine-mode Exception Program Counter
- `mstatus`: Machine-mode Status Register (initial value varies depending on XLEN)

2. **CSR Operation Type (CSROpType)**:
- `jmp`: CSR Jump Operation
- `wrt`: CSR Write Operation
- `set`: CSR Set Operation
- `clr`: CSR Clear Operation
- and corresponding immediate value versions (`wrti`, `seti`, `clri`)

3. **Special Instruction Detection**:
- `isEcall`: Detected context call instruction
- `isMret`: Detected machine-mode return instruction
- `isEbreak`: Detected breakpoint instruction

4. **CSR Read/Write Logic**:
- Selects the CSR register to access based on `srcb` (CSR address)
- Determines the operation type (write/set/clear) based on `fuOpType`.
- Updates the corresponding CSR when the write enable (`csrWen`) is enabled.

5. **Exception Handling Logic**:
- When `ecall` occurs:
- Sets `mcause` to the machine-mode context to call the exception code.
- Saves the current PC to `mepc`.
- When `mret` is executed, returns the address stored in `mepc`.

6. **Output Logic**:
- Outputs the selected CSR value during normal CSR operation.
- Outputs `mtvec` (trap handler entry point) during `ecall`.
- Outputs `mepc` (return address) during `mret`.
- Asserts the `jmp` signal when a jump is required.

7. **CSR Address Constants**:
- Defines all standard RISC-V CSR addresses (machine mode, user mode, and supervisor mode).
- Including status registers, trap vectors, exception handling-related CSRs, etc.

8. **Exception and Interrupt Priority**:
- Defines exception type codes and priorities
- Defines interrupt types and priorities
